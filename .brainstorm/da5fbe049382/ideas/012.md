# Idea 12: Monorepo Package Boundary Referee

Take real monorepos (Babel, Next.js, Turborepo, NX-managed projects, Rust workspace crates). Introduce a change in one package and ask: does this violate any package boundary? Should this be a breaking change for consumers? Ground truth from actual semver bump decisions and CHANGELOG entries in the real repo. The agent must understand package.json/Cargo.toml workspace configs, internal vs public API surfaces, re-exports, and the implicit contracts between sibling packages. Scoring: correct classification of change impact (none/patch/minor/major) per affected package. Tests understanding of organizational code boundaries, not just call graphs.
