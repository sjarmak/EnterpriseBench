# Idea 5: API Contract Boundary Analysis

Take OSS projects with well-defined API boundaries (gRPC services in the CNCF ecosystem: Envoy, Istio, CoreDNS, etcd). Each task presents a proposed proto/OpenAPI change and asks: which downstream consumers break? The agent must trace API surface through proto definitions, generated code, client libraries, and integration tests across 3-5 repos. Ground truth: the actual breakage found in the real PR review. Structurally different because the 'code' being understood is an interface contract that spans multiple repos through generated artifacts, not source code someone wrote.
