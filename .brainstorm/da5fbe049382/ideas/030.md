# Idea 30: Observability Instrumentation Gap Analysis

Take OSS projects that added observability after initial development (OpenTelemetry adoption PRs in real projects). Present the codebase BEFORE instrumentation was added. Task: identify where traces, metrics, and logs SHOULD be added for production-grade observability. Specifically: which functions need spans, which operations need latency histograms, which error paths need structured logging. Ground truth: the actual instrumentation PR. Unique: this isn't about understanding what the code DOES — it's about understanding what the code NEEDS operationally. The agent must reason about runtime behavior, failure modes, and SRE concerns from static code. Different cognitive axis than all prior ideas.
