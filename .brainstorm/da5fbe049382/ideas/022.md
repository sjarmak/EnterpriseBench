# Idea 22: Multi-Repo Refactor Orchestration Planning

Take real OSS refactors that spanned multiple repos (e.g., gRPC API version bumps across CNCF projects, or Python 2→3 migrations in interconnected packages). Give the agent all repos at the pre-refactor state. Task: produce a TOPOLOGICALLY SORTED execution plan — which repos change first, what's the safe ordering to avoid breaking intermediate states, which changes can be parallelized. Ground truth: the actual chronological order of the real PRs. Scoring: plan validity (no broken intermediate states) + efficiency (parallelism). This measures understanding of cross-repo ORDERING CONSTRAINTS, not just what changes.
