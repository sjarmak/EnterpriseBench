# Idea 7: Release Impact Changelog Generation

Given a large OSS repo at two tagged releases (e.g., Kubernetes 1.28→1.29), the agent must produce a user-facing changelog that correctly categorizes every breaking change, deprecation, new feature, and security fix. Ground truth: the actual release notes written by maintainers. Scoring: coverage of real changes + absence of hallucinated changes + correct categorization. This tests understanding at the DIFF level — not one file, but hundreds of commits synthesized into a coherent narrative. Enterprise teams do release impact analysis constantly.
