# Incident: Service Mesh 503 Errors After Istio Upgrade

## Alert

After upgrading Istio from 1.19 to 1.20, intermittent 503 errors are observed for a subset of HTTP requests in the service mesh. The affected requests target VirtualServices that use header-based routing with regex matching.

## Symptoms

- 503 UC (Upstream Connection) errors in Envoy access logs
- Only affects routes using regex header matching in VirtualService
- Stable routes with exact header matching are unaffected
- The issue appeared immediately after the Istio control plane upgrade
- Envoy sidecars show the routes in their config dump, but matching fails

## Investigation So Far

The team has identified that the issue is related to regex header matching:

1. The VirtualService uses `headers: { x-canary: { regex: "^true$" } }` matching
2. Istio 1.20 changed the default regex engine configuration for Envoy
3. The Envoy route configuration is generated by Istio Pilot's route translation code
4. Envoy's RouteMatch uses `safe_regex` with a configurable engine

## Environment

- Istio 1.20.0 at `/workspace/istio/`
- Envoy v1.28.0 at `/workspace/envoy/`
- Focus on Istio Pilot route translation (Go) and Envoy router (C++)

## What I Need

1. **Root Cause**: Find where Istio Pilot translates VirtualService HTTPRoute header matching into Envoy route configuration. What changed in 1.20 that affects regex engine configuration?

2. **Error Chain**: Trace the full data flow: VirtualService config -> Pilot route translation -> Envoy route config -> regex header matching -> 503 response.

3. **Affected Components**: Which service mesh components and VirtualService patterns are affected?

4. **Remediation**: How should the configuration be fixed?
